# 03. 落とし穴カタログ(答え合わせ用) — 11カテゴリ85パターン

各パターンは以下の4点で記述する。
**現象**=何を作り込んだか / **MF**=メインフレームでなぜ正しく動く(動いてしまう)か /
**移行**=移行後の典型的な壊れ方(アベンド・結果相違・性能劣化) / **場所**=埋込資産。

難易度: ★=静的解析で発見可 / ★★=データと突合しないと発見不可 / ★★★=実行・並行検証でしか顕在化しない

---

## A. 文字コード・DBCS系(11)

**A-01 SO/SI 可変長氏名** ★★
現象: PIC X(40) の氏名欄に SO(0x0E)/SI(0x0F) 挟みの DBCS と SBCS が混在。バイト数≠文字数。
MF: IBM 漢字(EBCDIC DBCS, CCSID 930/939 系)では SO/SI 込みで桁数勘定するのが正。BMS も DBCS 混在フィールドを表示可能。
移行: UTF-8 変換で SO/SI が消え桁位置が崩壊。固定位置 substring(A-06)が別項目を切り出す。
場所: JYUMAST 氏名/住所、RAJYU01C、RBSTM02C 印字編集。

**A-02 SI 欠落レコード(不整合 DBCS)** ★★★
現象: データ中に SO で開いたまま SI が無い(または奇数バイト)レコードを混入。
MF: バッチは項目をバイト列として素通しするため異常終了しない。印字時に化けるだけで運用上「既知の汚れ」。
移行: コード変換(EBCDIC→UTF-8)ツールが例外・置換文字化・後続項目まで巻き込み破壊。
場所: JYUMAST 12件、KENFILE 備考 5件(件数・キーは 04_data_sekkei.md に明記)。

**A-03 外字(ユーザ定義文字)** ★★
現象: 氏名に外字コード(IBM 漢字 DBCS-Host のユーザ定義文字域 0x6941〜0x89BD)を使用。「髙」「﨑」等の異体字+純粋外字。
MF: サイト外字テーブル+専用プリンタで正しく印字。
移行: 変換表が無く「〓」化・例外。氏名突合バッチ(RBSHU01C)で不一致→消込漏れという**業務結果の相違**に波及。
場所: JYUMAST 約120件、外字変換は ASM(RASMKNJ)のみが知っている(G-05 と連動)。

**A-04 半角カナと英小文字の CCSID 衝突** ★★
現象: カナ氏名・全銀レコードに半角カナ(ｶ)ﾄｳｻﾞｲｼｮｳｼﾞ 等)。EBCDIC カナ(CCSID 290/930 系)では英小文字とカナが同一コード域を奪い合う。
MF: カナ端末・カナプリンタ前提で一貫。英小文字は業務上使わない規約で回避。
移行: 変換テーブル選択(930系/939系)を誤ると英小文字⇔カナが化ける。プログラム内リテラル比較('ｱ' 等)も対象。
場所: JYUMAST カナ氏名、全銀収納ファイル、RBSHU01C 内リテラル。

**A-05 全角スペース初期値** ★★
現象: DBCS 項目の未使用部が全角スペース埋めのレコードと半角スペース埋めのレコードが世代により混在。
MF: `IF KANJI-NAME = SPACE` は半角スペース埋めのみ真。全角埋めは「入力済み」と誤判定されるが、後続処理が偶然無害で顕在化せず。
移行: 変換・トリム仕様の差で判定が反転し、未入力扱い/入力済み扱いが変わる。
場所: RAJYU01C、RBSTM01C。

**A-06 DBCS 固定バイト位置切り出し** ★★
現象: 氏名 X(40) の先頭12バイトを「姓」とみなす REDEFINES。SO/SI 込みバイト位置なので文字境界と無関係。
MF: 印字ではたまたま整う(データ規約で姓は5文字以内)。規約外データは化けるが放置。
移行: 文字単位 substring に置き換えると**結果が変わる**(正しくなってしまい現新不一致)。
場所: RBSTM02C、コピー句 RCJYUREC。

**A-07 EBCDIC ソート順依存** ★★★
現象: カナ氏名+数字混在キーで SORT し、ブレーク処理(コントロールブレーク)の前提に。EBCDIC は 空白<英小<カナ<英大<数字 の順。ASCII/Unicode では数字が英字より前。
MF: 正しい帳票順(数字枝番が後ろ)。
移行: 並び順が変わりブレーク単位が崩れ、**小計が別グループに紛れる**。ソート指定自体は無傷なので静的解析では見えない。
場所: JCL RJM020(SORT)+RBSTM01C のブレーク処理。

**A-08 PIC G と PIC X の二重定義** ★
現象: 同一領域を PIC G(20) USAGE DISPLAY-1 と PIC X(40) で REDEFINES し、処理により使い分け。
MF: DBCS 対応コンパイラでは正当。G 側は SO/SI なし純 DBCS、X 側はバイト操作用。
移行: G→String 変換と X→byte[] 変換で同一領域に2つの型が生じ、変換器が不整合コードを生成しがち。
場所: RCJYUREC、RAJYU01C。

**A-09 濁点分離カナ・機種依存文字** ★★
現象: 「カ゛」(カ+結合濁点相当の2文字格納)、丸数字・ローマ数字相当の私用コード。
MF: 端末・プリンタが解釈し見た目は正常。
移行: NFC 正規化の有無で文字数・比較結果が変化。氏名マッチング(RBSHU01C)の結果相違。
場所: JYUMAST 約40件。

**A-10 X 項目と COMP-3 の同居 REDEFINES** ★★
現象: 予備領域 X(8) を、レコード種別により COMP-3 金額 / 文字フラグ('F' 埋め)として二重利用。
MF: 種別で分岐するため矛盾しない。'F' のバイト列は偶然パック10進として読める(F は数字ゾーン)。
移行: 一括コード変換(EBCDIC→ASCII)でパック側が破壊される、または型推論が片方に寄せて他方が壊れる。
場所: RYOFILE 予備域、RBRYO02C。

**A-11 漢字リテラルの16進直書き** ★★
現象: ソースに日本語を直接書けなかった時代(端末・コンパイラ・コード系の制約)の名残で、帳票見出し・画面固定文言・エラーメッセージを `VALUE X'0E45C34AF8...0F'` の16進定数で保持。コメントに読み仮名があるものと**無いもの**が混在。姉妹現象として、16進すら避けてローマ字化したメッセージも共存(RUTLDTC の 'FUNC FUKASEI' が実装済の実例)。
MF: IBM 漢字コードポイントの直書きなので、意図した字が確実に出る。
移行: ソースコード変換ツールは16進リテラルを**文字として認識せず無変換** — Unicode 環境で別文字化・破壊。意図復元には IBM 漢字コード表と外字表(A-03/G-05)が必要で、16進の中に SO/SI そのものが含まれるため A-02 系の不整合も混入可能。
場所: RBSTM02C(帳票見出し)、RAMEN01C(画面固定文言)、うち3件は読み仮名コメント無し。

## B. データ表現系(12)

**B-01 ゾーン10進の符号オーバーパンチ** ★★
現象: 全銀振替結果の金額訂正レコードで末尾桁 'J'〜'R'(=−1〜−9)により負値表現。PIC S9(9) DISPLAY。
MF: ゾーン10進の正規表現。COBOL は普通に負数として演算。
移行: テキストとして CSV 化・Java String→int 変換で例外、または「12345J」を捨てて金額 12345 と誤読 → **消込金額の相違**。
場所: 全銀 SHUIN ファイル、RBSHU01C。

**B-02 数字項目のスペース** ★★★
現象: PIC 9(5) の検針員コードにスペース(未設定)が混入。
MF: NUMPROC(NOPFD)/ゾーン部無視の実装では 0 相当で通過。IF NUMERIC を通さない箇所で ADD しても S0C7 にならない設計値のみ使用。
移行: Java の parseInt が例外、あるいは 0 埋め変換で「検針員 00000」が新規に出現し集計結果相違。
場所: KENFILE 約200件、RBKEN02C(NUMERIC チェックを**していない**パス)。

**B-03 COMP-3 不正ニブル(符号 F/C 混在)** ★★
現象: 同じ正数でも旧システム由来レコードは符号 F(無符号)、現行は C(正)。比較キーに使用。
MF: 演算は同値。ただし `IF A = B` のバイト比較箇所では不一致 → それを前提にした「旧データ判定」ロジックが存在。
移行: BigDecimal 化で両者同値になり、**旧データ判定が全滅**(判定できなくなる)。
場所: KYKMAST 単価世代、RBCNV01C。

**B-04 BINARY の TRUNC 依存** ★★
現象: PIC 9(4) COMP のカウンタを 9999 超まで回し、TRUNC(OPT)/(BIN) 前提で 65535 まで保持。
MF: コンパイラオプションで合法的に動く(現場あるある: 「STD にすると壊れるので触るな」コメント付き)。
移行: 10進4桁に正しく切り詰める処理系へ移すと値が変わる。
場所: RBGET01C 通番採番。

**B-05 REDEFINES マルチフォーマット VB レコード** ★
現象: 検針受信ファイルはヘッダ(H)/明細(D)/計器交換(K)/トレーラ(T)の4種を一つの 01 に REDEFINES。RECFM=VB。
MF: 標準的技法。
移行: 変換器がフォーマット判別ロジックを取り違え、K レコードを D として処理(F-04 と複合)。
場所: RCKENIN コピー句、RBKEN01C。

**B-06 OCCURS DEPENDING ON + 末尾ゴミ** ★★
現象: 明細レコードの繰返し数 n(最大24: 30分値の一部送信)を ODO で定義。n を超えた領域に前レコードの残骸が残る。
MF: ODO 範囲外は参照しないので無害。
移行: 固定長展開(ODO を最大長でフラット化)する変換器が**残骸を実データとして読む**。
場所: SMTIN 30分値受信、RBAUD02C。

**B-07 LOW-VALUES と SPACES の初期化すれ違い** ★★
現象: キー項目を LOW-VALUES 初期化する旧世代と SPACES 初期化する新世代が混在し、START ... KEY >= で「全件先頭から」を意図。
MF: EBCDIC で LOW-VALUES(0x00)< SPACE(0x40)< '0'(0xF0) のため両方とも先頭に到達。
移行: ASCII では SPACE(0x20)< '0'(0x30) は保たれるが、DB 移行で NULL/空文字/最小キーの扱いが割れ、**開始位置がずれて先頭数件を読み飛ばす**。
場所: RAKYK00C、RBKYK01C。

**B-08 桁あふれ循環を仕様として利用** ★
現象: 6桁伝票番号が 999999 の次に 000001 に戻る前提(ON SIZE ERROR なし、COMPUTE の黙示切捨て)。
MF: 黙示の切捨てで循環し、既存運用は重複期間を跨がない。
移行: 例外送出型の演算に置き換わり月替りにアベンド。
場所: RBRYO02C。

**B-09 部分参照(参照修飾)による定義なき部分項目** ★★
現象: 供給地点特定番号 X(22) を (3:2)=地区コード、(21:2)=チェックデジットのように参照修飾で切り出す。この内部構造はコピー句のどこにも定義されていない。
MF: 合法かつ「定義を増やさない」省エネ作法として多用された。
移行: 項目単位の影響調査・データリネージ解析から部分項目が漏れる。桁位置直書きのため桁数変更に脆く、DBCS 混在項目では文字単位/バイト単位の差(A-06)により切り出し結果自体が変わる。
場所: RUTLDTC(LK-DATE-IN(1:6)、実装済)、RBSHU01C(全銀レコード切り出し)、RASMCHK 呼出部。

**B-10 COMP-1/COMP-2(16進浮動小数点)による近似計算** ★★★
現象: 負荷率・力率・按分係数の計算に COMP-2 を使用し、`IF W-RITU > 0.85` のような閾値比較を行う。z の伝統的浮動小数点は **HFP(基数16)** であり、IEEE754(基数2)と仮数の刻み・丸めが異なる。
MF: HFP で一貫している限り再現性があり正しい。
移行: IEEE754 環境では最下位桁の近似値が変わり、**閾値ちょうど付近のデータだけ判定が反転**する。演算順序の結合則も崩れるため総和も微差。データ側に境界値需要家を意図的に混入し、この反転を必ず顕在化させる(dirty_ledger 登録)。
場所: RBGET01C(負荷率)、RBAUD01C(30分値の平均・力率)、境界値データは 04_data_sekkei.md #16。

**B-11 MOVE CORRESPONDING(名前一致の暗黙転記)** ★★
現象: 集団項目間の転記を MOVE CORRESPONDING で行う。転記対象は「その時点で名前が一致する項目」で決まり、ソース上に列挙されない。コピー句の項目追加・改名(H-01 の揺れと連動)で転記対象が**静かに増減**してきた歴史を持つ。
MF: 合法。74 時代の省力作法として多用された。
移行: 明示転記への展開時、**現在の**コピー句で展開すると過去の改修時点とは違う対応になる恐れ。名前一致という仕様自体が命名揺れ(H系)と干渉し、意図した対応表の復元が困難。
場所: RAJYU01C、RBRYO02C。

**B-12 予備領域の枯渇と型圧縮リフォーム** ★★★
現象: 初版(1985)で確保した FILLER X(30) が保守のたびに侵食され残り X(2)。レコード長・キー位置は影響範囲が広すぎて変更不能のため、項目追加の際に**既存項目の型属性を変更してバイトを捻出**してきた — 金額 PIC 9(9) DISPLAY(9byte)→ S9(9) COMP-3(5byte)で4byte 確保、日付 9(8)→COMP-3 5byte、連番の BINARY 化。さらに型圧縮**前**の旧レイアウトのレコードが同一ファイル内に残存し、レイアウトバージョン項目は無く、**適用開始日による暗黙判別**で新旧を読み分ける。
MF: 変換プログラム+暗黙判別+口伝の旧レイアウト知識で整合している。レイアウト定義書は最新版しか保守されていない。
移行: (1)「なぜこの項目だけ COMP-3 か」というバイト捻出史が失われ、DB 設計で素直に正規化→旧レコードが読めない。(2) 暗黙判別条件を見落とし、旧レコードを新レイアウトで解釈して桁ずれ全滅。(3) 残り FILLER X(2) を「未使用」と削除し、拡張余地と判別の手掛かりを破壊。H-05 が**台帳に無い**非公式転用であるのに対し、本項は**正式な転用が積層した末の枯渇**という対の関係。コピー句には増改築履歴コメント(昭和〜令和)を残し、地層を読ませる。
場所: RCKYKREC/RCJYUREC(コピー句)、RBCNV01C(新旧変換)、旧レイアウトレコードは KYKMAST の約8%(dirty_ledger とは別に正規仕様として台帳化)。

## C. 制御構造系(8)

**C-01 ALTER / GO TO** ★
現象: 検針票編集サブ(RBSTM02C)の頁制御を ALTER で書換える 1980年代スタイル。
MF: 合法。
移行: 変換器の鬼門。状態機械への正確な再構成が必要で、誤ると**改頁位置がズレた帳票**を静かに出力。
場所: RBSTM02C。

**C-02 PERFORM THRU + 範囲外 GO TO** ★
現象: PERFORM A THRU A-EX の内部からエラー時に範囲外の ABEND-RTN へ GO TO(戻らない)。
MF: 合法(スレッドは戻り先を失うが STOP RUN で終わるので実害なし)。
移行: 構造化変換で「戻ってくる」コードになり、エラー時に**処理が継続してしまう**。
場所: RBKEN01C、RBRYO01C。

**C-03 セクション・フォールスルー** ★★
現象: PERFORM されないセクションが直前セクションから流れ落ちる形で実行される(MAIN の末尾に置かれた「後処理」)。
MF: 合法。GO TO で飛び越すパスもあり、実行経路はデータ依存。
移行: セクション=メソッドと機械的に切ると**後処理が呼ばれない経路**が生まれる。
場所: RBTAI01C。

**C-04 サブプログラム内 STOP RUN** ★
現象: 共通サブ RUTLDTC が致命エラー時に GOBACK でなく STOP RUN。呼出元の後始末(ファイルクローズ、リスタート記録)がスキップされる。
MF: それ自体が「異常時は全部止める」仕様として機能。
移行: 例外スロー化すると呼出元の catch が後始末を実行し、**リスタートポイントの意味が変わる**。
場所: RUTLDTC、全バッチ。

**C-05 NEXT SENTENCE の罠** ★
現象: END-IF 混在ソース内の NEXT SENTENCE(ピリオドまで飛ぶ)により、意図より遠くへ飛ぶ箇所が1箇所(ただし現データでは経路に入らない)。
MF: 潜在バグだが未発火。**「正しく動いているが正しくないコード」**の代表。
移行: AI/人手が「意図」を推測して修正すると現新比較で不一致(正しくなってしまう)。仕様として残すか否かの判断問題。
場所: RAKEN01C。

**C-06 GO TO DEPENDING ON + フラグ書換え** ★
現象: 処理モード(1:通常 2:再計算 3:概算)を GO TO DEPENDING ON で分岐、内部でモードを書換えて再突入。
MF: 合法。
移行: switch 化の際に再突入ループが失われがち。
場所: RBRYO01C。

**C-07 WORKING-STORAGE 残存値依存** ★★★
現象: バッチ RBSTM01C が RBSTM02C を CALL し、02C 側の前回呼出しの WS 残存値(頁番号・累計)を「引き継ぎ仕様」として利用(INITIAL 属性なし)。
MF: 静的呼出しの WS は残存するのが既定。
移行: インスタンス生成毎初期化の言語に移すと**頁番号が毎回1に戻る**。
場所: RBSTM01C/02C。

**C-08 上方向 GO TO(後方分岐によるループ形成)** ★★
現象: PERFORM UNTIL ではなく、既出の段落先頭へ戻る GO TO でループを形成。脱出条件が複数箇所に分散し、ループ本体の途中からも脱出する。
MF: 合法で、当時の標準作法。前進 GO TO と混在してスパゲッティを構成する。
移行: 構造化変換(while/for への再構成)の最難点。終了条件の取り違え・無限ループ化・反復回数の±1ズレが典型で、静的には正しく見えるのに**特定データでのみ結果が変わる**。
場所: RUTLDTC(UNS-020/UNS-030、実装済 3箇所)、RBKEN01C(受信レコード読み戻り)、RBRYO01C(再計算モード再突入: C-06 と複合)。

## D. ファイル・JCL系(10)

**D-01 FILE STATUS 未チェック** ★★
現象: 単価ファイル OPEN の FILE STATUS を見ず、READ AT END だけで制御。ファイル欠落時は0件として続行し**プログラム内テーブル(F-03)の単価で計算**。
MF: これが「単価ファイル無しでも動く」フェイルセーフとして運用に組み込まれている。
移行: OPEN 失敗を例外にすると随時ジョブが軒並み異常終了(または逆に、内蔵単価へのフォールバックが失われ古い単価適用が発覚)。
場所: RBRYO01C。

**D-02 DD DUMMY を仕様として利用** ★★
現象: 日次 JCL で再計算入力を DD DUMMY にし「0件=再計算なし」を表現。月次のみ実データセットを割当て。
MF: 標準技法。
移行: ファイル存在必須の環境で DUMMY 相当の表現を忘れ、日次が異常終了。
場所: RJD030。

**D-03 COND= の逆論理と IF/THEN 混在** ★
現象: `COND=(4,LT)`(=RC が 4 より大きければ**飛ばす**)と新しい IF RC > 4 THEN が同一 JCL に混在。さらに「RC=4 が正常、RC=0 は対象0件」という**リターンコード意味逆転**(D-09)と組合せ。
MF: 運用手順書どおりに動く。
移行: スケジューラ移行時に条件を1つでも読み違えると、**エラー時に後続が走る/正常時に止まる**。
場所: RJD010〜RJD090、RJM010。

**D-04 GDG 世代番号運用** ★
現象: バックアップ(+1)、当日参照(0)、前日比較(-1)。月次で LIMIT 超過削除。カタログ競合時の再実行手順が前提。
移行: 世代概念のないストレージへの写像ミス(常に最新を読む等)で**前日比較が同日比較**になる。
場所: RJD090、RJM030、RBGET01C。

**D-05 SORT E15/E35 出口** ★★
現象: 検針データの SORT に COBOL 出口 E15(入力選別: テストレコード除去)・E35(出力編集: 部門コード付与)。**ビジネスロジックが SORT 出口に埋没**。
MF: 標準技法。
移行: SORT をツール置換すると出口が失われ、テストレコード(F-05)が本番集計に混入。
場所: RJD020、RXSRT15C/RXSRT35C。

**D-06 DISP=MOD 追記** ★★
現象: エラーレコード累積ファイルを DISP=MOD で追記。再実行時は**手順書に従い事前削除**が前提。
移行: 再実行の冪等性がなく、リラン時に**二重計上**。ジョブ管理移行の定番事故。
場所: RJD020。

**D-07 RDW(可変長レコード記述語)前提コード** ★
現象: VB ファイルをレコード長フィールド込みで扱い、RECORD VARYING の LENGTH 値から明細数を逆算。
移行: RDW を落とす転送(FTP ASCII 等)でレコード境界喪失。
場所: RBKEN01C、SMTIN。

**D-08 ユーティリティ制御文の中のロジック** ★
現象: IDCAMS REPRO の SKIP/COUNT、IEBGENER の GENERATE/RECORD FIELD= による項目並べ替え=実質的な変換ロジックが JCL 側に存在。
移行: 「JCL はただの起動」と見なす解析で**変換仕様が丸ごと欠落**。
場所: RJM040、RJA010。

**D-09 リターンコード意味逆転** ★★
現象: RBTAI01C は「停止候補あり=RC4 / なし=RC0」を返し、後続の停止指示ジョブは COND で「RC0 なら flush」。
MF: 設計どおり。
移行: 「RC0=正常続行」の思い込みで後続を常時実行 → **停止候補0件の日に空振り帳票**、逆条件なら停止漏れ。
場所: RBTAI01C、RJD060。

**D-10 DECLARATIVES によるエラー処理の隠蔽** ★★
現象: 入出力エラー処理が手続き部本文ではなく宣言部(DECLARATIVES / USE AFTER STANDARD ERROR PROCEDURE)に置かれ、本文だけ読むと「エラー処理なし」に見える。一部の宣言部はエラーを記録だけして握りつぶし処理を継続する。
MF: 正当な言語機能で、当時の共通部品設計の定石。
移行: 例外処理機構への写像で宣言部が漏れる/本文に「エラー処理を追加」して二重化する。D-01(未チェック)と対をなす「実はチェックしているが見えない」パターン。
場所: RBSTM01C、RBBKU01C。

## E. 日付・暦系(6)

**E-01 和暦2桁+元号1桁** ★★
現象: 計器検定満期を 'S'/'H'/'R'+YY(2桁)で保持。ただし旧レコードは元号欄が**空白で YY のみ**(YY≧64→昭和、それ未満→平成、という暗黙ウィンドウ判定がコードに存在)。
MF: 判定ロジックが歴史的データと整合しており正しく動く。
移行: 西暦変換で暗黙ウィンドウを見落とすと**検定満期切れ計器の抽出漏れ**(法令業務の誤り)。
場所: MTRMAST、RUTLDTC、RBMTR01C。

**E-02 センチネル日付** ★★
現象: 00000000=未設定、99999999=無期限、99991231=解約予定なし、の3種が別意味で併存。
MF: 各プログラムが自分の見る項目の流儀だけ知っていて整合。
移行: DATE 型変換で例外、または一律 NULL 化で**「未設定」と「無期限」の区別が消える**。
場所: KYKMAST、JYUMAST、複数プログラム。

**E-03 YYMMDD ウィンドウイング** ★
現象: 検針日 PIC 9(6)。YY≧50→19xx。2000年対応時の暫定が恒久化。
移行: 2050年問題の温床、かつ変換器の基準年設定ミスで**過去日付が未来日付化**。
場所: KENFILE、RUTLDTC。

**E-04 検針月≠暦月** ★★
現象: 「6月分」=5月21日〜6月20日。月次集計 JCL は暦月 GDG、料金は検針月ベースで、**両者のズレを吸収する補正ロジック**が RBGET01C に存在。
移行: 補正を冗長コードと誤認して削除すると月次計が合わない。
場所: RBRYO01C、RBGET01C。

**E-05 独自うるう年判定(100年ルール欠落)** ★
現象: 4で割り切れれば閏年、のみ(1900/2100 誤判定)。現運用期間では正しい。
MF: 未発火の潜在バグ。
移行: 標準日付ライブラリ化で挙動が「直る」→現新比較で将来日付テストのみ不一致。仕様凍結か修正かの判断問題(C-05 と同類の「正しく動く誤り」)。
場所: RUTLDTC。

**E-06 パック10進の日付演算** ★
現象: 日付を COMP-3 の通算日(1980/1/1 起点の独自シリアル値)で加減算する箇所と YYMMDD 直接演算の箇所が混在。
移行: 起点の異なる2系統を統一する際の±1日ズレ。
場所: RUTLDTC、RBTAI01C(督促起算日)。

## F. 業務ロジック系(8)

**F-01 端数処理の混在** ★★★
現象: 基本=切捨て / 従量各段=四捨五入 / 燃調=銭保持→最後に切捨て / 割引=切上げ。中間結果の保持桁も項目ごとに異なる。
MF: 制度どおりで正しい。
移行: BigDecimal 化の際に丸めモード・丸めタイミングを1箇所でも標準化すると**1円ズレが数百件/月**発生。並行比較(現新突合)の主戦場。
場所: RBRYO01C(答え: 丸め箇所は7箇所、順序固定)。

**F-02 消費税の歴史的経緯** ★★
現象: 税率テーブル(3/5/8/10%+経過措置)と、内税化以前の**外税レコード**が RYOFILE 旧世代に残存。再計算(モード2)は旧税率で再現計算する必要がある。
移行: 「現行税率のみ」実装で過去分再計算が不一致。
場所: RBRYO01C、TANKA、RYOFILE。

**F-03 単価の二重管理** ★★
現象: 単価はプログラム内テーブル(RBRYO01C の WS)と TANKA ファイルの二重持ち。随時ジョブ RBTBL01C で同期するが、**内蔵テーブルが正**(D-01 のフォールバック)。
MF: 運用ルールで不整合を回避。
移行: 「ファイルが正」と誤認して内蔵テーブルを捨てると、ファイル欠落時挙動が変わる。仕様理解の誤りやすさの代表。
場所: RBRYO01C、RBTBL01C。

**F-04 マイナス使用量=計器交換の暗黙仕様** ★★★
現象: 今回指示数<前回指示数のとき、(計器満了値−前回)+今回 で使用量算出。交換レコード(K)が来ない旧型計器では**マイナス差分だけが交換の証拠**。
MF: 検針員の運用と一体で正しく動く。
移行: 「マイナスはエラー」と実装すると交換月の請求が全滅。仕様書に書かれていない**現場仕様**の典型。
場所: RBKEN02C、RBRYO01C。

**F-05 マジックナンバー契約** ★★
現象: 契約種別 '99'=社内テスト用、需要家番号 9000000000 番台=ダミー。集計・請求から**暗黙に除外**(比較のハードコード散在)。
移行: 除外条件を1本でも移し漏れるとテストデータが本番集計に混入(D-05 と複合)。
場所: 全バッチに散在(答え: 除外比較は11箇所)。

**F-06 バッチ実行順序が仕様** ★★★
現象: 収納突合(D050)→滞納判定(D060)の順序前提。同日再実行時は D050 から。順序を守らないと二重消込は**起きないが督促が1日早く出る**。
移行: ジョブ並列化・順序最適化で微妙な業務結果相違。
場所: RJD050/RJD060、運用手順書(docs 内に再現)。

**F-07 複合 COMPUTE の中間結果桁依存** ★★★
現象: `COMPUTE KIN = SIYORYO * NENCHO-TANKA / 100 + WARIBIKI * RITU` のような乗除混在の一文複合式。中間結果の保持桁・スケールはソースに現れず、IBM Enterprise COBOL の中間結果規則(ARITH(COMPAT)=最大30桁、除算位置により小数部が決まる)に暗黙依存。除算が式の途中に来る箇所では**暗黙の切捨てが仕様の一部**になっている。
MF: コンパイラ規則どおりに一貫して動く。
移行: BigDecimal 化で「数学的に正しい」精度にすると結果が変わる(良くなってしまい現新不一致)。逆に double 経由だと別方向にズレる。式を分解・並べ替えた瞬間に中間桁が変わるため、**変換器のリファクタリングそのものがバグ源**。RUTLDTC の `(HDK-NEN - 1) / 4` はこの最小例(整数除算切捨て依存)。
場所: RBRYO01C 燃調・段階料金・割引の複合式(答え: 除算先行3箇所・切捨て依存5箇所)、RBGET01C。

**F-08 按分・累積丸めの残差寄せ** ★★★
現象: 契約変更月の基本料金日割りで、日数按分額を明細ごとに円未満丸めし、**合計との差(残差)を最終明細に寄せる**。月次集計側は「明細合計=請求額」の一致を前提に突合する。
MF: 残差寄せがあるから帳尻が常に合う。
移行: 按分を素直に実装(全明細均等丸め)すると合計が±数円ズレて突合バッチがエラー、または残差寄せの実装順序(ソート順)が変わると**寄せ先の明細が変わり**明細単位の現新比較が不一致。丸め誤差は1件1円でも 6万契約×12ヶ月で累積し、月次集計の突合で必ず顕在化する設計とする。
場所: RBRYO01C(日割り按分)、RBGET01C(月次突合)、RBRYO02C。

## G. 環境・ミドルウェア系(8)

**G-01 EIBCALEN 分岐** ★
現象: COMMAREA 長 0=初回表示 / >0=再入、の疑似会話定石に加え、**長さの値そのもの**(旧画面 120/新画面 200)で新旧遷移元を判定。
移行: DTO 化で長さ情報が消え、遷移元判定が崩壊。
場所: RAKYK02C、RAMEN01C。

**G-02 CWA/TS キューの共有状態** ★★
現象: 端末 ID 連結名の TS キューにページング状態、CWA に単価キャッシュ。
移行: ステートレス Web 化で最も設計変更を要する箇所。TS 削除タイミング(PF3 時のみ)を移し漏れるとキュー残骸で**別端末に前ユーザの頁が見える**。
場所: RAKYK00C、RAAUT00C。

**G-03 IMS SSA の DBCS 混在キー** ★★
現象: 需要家 DB のルートキーは需要家番号だが、副次索引に**カナ氏名(半角カナ)**。SSA 組立てで CCSID 依存比較。
移行: RDB 化時の照合順序(A-07 と同根)+カナ正規化で検索ヒット件数が変わる。
場所: JYUDBD、RAAUP01C。

**G-04 MQ メッセージ内の COMP-3** ★★★
現象: デマンド要求メッセージは文字+COMP-3+BINARY 混在の固定レイアウト。MQ の CCSID 自動変換(CONVERT)を**バイナリ部があるため無効化**し、アプリで部分変換。
MF: 設計どおり。
移行: メッセージを JSON 化する際の桁・符号・スケール写像ミス、または自動変換 ON で**パック10進が文字変換され破壊**。CardDemo オーソリ相当の目玉罠。
場所: RAAUP00C、RCDMDMSG コピー句。

**G-05 アセンブラ共通部品** ★★
現象: 外字・カナ変換(RASMKNJ: TR 命令+256/DBCS 変換テーブル)とチェックデジット(RASMCHK: CVB/CVD、供給地点特定番号22桁のモジュラス計算をパック分割で実装)。COBOL から CALL。
MF: 30年間無事故の「触ってはいけない」部品。
移行: 移行対象一覧から漏れる/変換テーブル(データ)を移さない、が定番。**変換表こそが外字仕様書**(A-03 の答えはここにある)。
場所: app/asm/、呼出元多数。

**G-06 Db2 FOR BIT DATA に DBCS** ★★
現象: 警報履歴表の需要家名列を CHAR(40) FOR BIT DATA で定義し、EBCDIC バイト列のまま格納(歴史的経緯: 変換化けを避けた)。
MF: 読み書き同一系なので無害。
移行: Unicode DB へ移すと**この列だけ二重変換/無変換**でバイナリ文字列化。
場所: db2/TALMHIST.ddl、RAAUP01C、RBAUD01C。

**G-07 未初期化領域のゼロ前提** ★★★
現象: VALUE 句のない WORKING-STORAGE のカウンタ・集計域を、初期化せずに ADD で使い始める。また RAKYK02C は旧画面(COMMAREA 120byte)から呼ばれた際も新画面用の項目(オフセット120以降)を参照する(EIBCALEN 超過参照)。
MF: バッチ region の新規獲得ストレージや CICS の GETMAIN(INITIMG) が事実上ゼロ埋めのため「たまたま常に」正しく動く。COMMAREA 超過参照もストレージ保護に掛からず化けた値を読むだけで、その経路では値を使わないため無害。
移行: 言語・ランタイムの初期値規則が変わると集計値が不定(ゴミ加算)или実行毎に結果が変わる再現困難バグに化ける。COMMAREA の DTO 化では超過参照が即例外。**「OS が黙って守っていた仕様」**の代表格。
場所: RBKEN02C・RBGET01C(VALUE なし集計域)、RAKYK02C(EIBCALEN 超過: G-01 と複合)。C-07(残存値依存)が「前回値が残る」前提なのに対し、本項は「初回がゼロ」前提という対の関係。



**G-08 翻訳・実行時オプション依存の意味論** ★★★
現象: 同一ソースでも翻訳オプションと LE ランタイム設定で挙動が変わる要素に依存する。NUMPROC(PFD/NOPFD) による符号処理と NUMERIC クラス判定の差(B-02 の「スペース入り数字が通る」は NOPFD 前提)、ZWB(符号付きゾーン項目と文字の比較)、TRUNC(B-04)、SSRANGE 無効(J-03)、OPTIMIZE(「OPT にすると結果が変わるので触るな」の警告コメント付き)、LE の TRAP/CEE 設定によるデータ例外・ゼロ除算時の継続/アベンド差(J-05 連動)。しかも**翻訳オプションはプログラムごとに翻訳 JCL/PROC に散在して不統一**であり、その事実は翻訳記録簿(K-04)にしか現れない。
MF: 各プログラムは「自分のオプション」の下で30年間一貫して動いてきた。
移行: **ソースだけ読んでも動作が確定しない** — 移行仕様は「ソース+翻訳オプション+ランタイム設定」の三点セットで初めて定まる。オプション差を見落とすと、同じ書き方の2プログラムが移行後に片方だけ挙動が変わる、という原因究明が極めて困難な不一致を生む。B-02/B-04/J-03/J-05 は本パターンの個別事例であり、採点では翻訳記録簿のオプション列との突合まで要求する。
場所: app/jcl/ 翻訳 JCL・PROC(不統一を意図的に構成)、compile_ledger.csv(オプション列)、RBKEN02C(NUMPROC(NOPFD) 前提)、RBSHU01C(ZWB 前提)、RBGET01C(TRUNC(OPT) 前提)。

場所: RUTLDTC(適用済)、全プログラム。

**H-05 予備域・FILLER の転用** ★★★
現象: レイアウト上の YOBI(予備)に停止理由コードが実運用で格納され、特定プログラムだけが REDEFINES で参照。
MF: 動く。レイアウト定義書には現れない「生きた項目」。
移行: コピー句だけを正とした移行で項目が消失。A-10 と連動。
場所: KYKMAST 予備域、RATEI00C(Phase 1 で実装)。

## I. 資産・重複系(5)

**I-01 コードクローン(改名複製と派生分岐)** ★★★
現象: 3層のクローンを作り込む。(1) Type-1/2: RAUSR00C〜03C はコピー起源で変数名だけ改名(H-01 と連動)。(2) Type-3(派生クローン): 照会系 RAKYK01C/RAMTR01C は共通祖先からの複製で、**片方にだけバグ修正が適用済み**(もう片方は同じ潜在バグを保持=修正漏れの再現)。(3) 共通化されていない共通処理: RUTLDTC が存在するのに、複数プログラムが日付検証を**ローカル複製**しており、しかも複製ごとに判定が微妙に違う(うるう年100年ルールの有無が混在: E-05 連動)。
MF: それぞれ独立に「動いている」ため問題にならない。
移行: クローン検出→共通化リファクタリングで**意図的差分(業務上の例外)まで消える**、逆に検出漏れで片方だけ修正され現新比較の不一致が別プログラムで発生。どの差分が意図でどれが修正漏れかの判定が意図復元の核心問題になる(答え: 差分台帳を docs 側に保持)。
場所: RAUSR00C〜03C、RAKYK01C/RAMTR01C、日付ローカル複製は RBTAI01C/RBSTM01C。

**I-02 デッドコード(到達不能・恒偽分岐)** ★★
現象: (1) 廃止契約種別 '03' の処理ブロック(該当データ0件)。(2) 恒偽分岐: OLD-SYS-FLG が初期値 'N' のまま更新箇所がなく、'Y' 側のブロックが死んでいる。(3) どこからも PERFORM/GO TO されない段落。ただし**準デッド**(年1回・過去再計算時のみ生きる分岐: F-02 の旧税率処理)を混在させる。
MF: 実行されないだけで無害。
移行: 「デッドコードは削除」の一律方針が準デッドを巻き込み、**年次処理・再計算で初めて発覚する欠落**を生む。到達可能性判定の演習(静的には C-03 フォールスルーが判定を妨害する)。
場所: RBRYO01C、RBTAI01C、RAKYK02C。

**I-03 コメントアウト堆積と旧仕様コメント** ★★★
現象: 旧ロジックがコメントアウトで大量残存(RBRYO01C に約150行)。日付入り改修メモ(「H09.3 単価改定対応」等)のうち数件は**その後の改修で実コードと矛盾**した記述のまま放置。
MF: コメントは実行に関与しない。
移行: AI による仕様復元がコメントを一次情報として採用し、**現行コードではなく旧仕様を「意図」として復元**する。意図への遡行(intent recovery)の最重要検証点であり、「コメント≠仕様、コードとデータが真実」を問う。矛盾コメントの位置は答え合わせ台帳に記録。
場所: RBRYO01C、RBSTM02C、RUTLDTC ヘッダの電算部メモ(既に一部実装済)。

**I-04 未参照資産と年次限定資産(棚卸しの罠)** ★★
現象: 旧版料金計算 RBRYO00C がソース・ロードモジュールとも残存。日次・月次のどの JCL からも呼ばれないが、**年次 JCL(RJY010: 年度末特別再計算)からのみ CALL される**。加えて、どこからも参照されないコピー句・PROC を数点混在。
MF: 資産管理台帳上は「現役」。年1回だけ本当に動く。
移行: 静的参照解析だけの棚卸しで「未使用」と判定して移行対象から除外→翌年度末に欠落発覚、という移行対象選定(disposition)の定番事故を再現。スケジューラ定義まで含めた到達性解析ができるかを問う。
場所: RBRYO00C(新設)、RJY010(新設)、未参照コピー句2本。

**I-05 デバッグ残骸(D欄・条件付き DISPLAY・USE FOR DEBUGGING)** ★★
現象: 標識領域 'D' のデバッグ行、デバッグフラグで制御された DISPLAY、USE FOR DEBUGGING 宣言部が残置。本番は翻訳オプション/実行時 PARM で無効化されているだけ。
MF: WITH DEBUGGING MODE を外せば D 欄はコメント扱い — 「消さずに眠らせる」文化。
移行: D 欄の扱いを誤ると**デバッグ文が本番で覚醒**(大量出力で性能崩壊・ログ汚染)、逆に一律削除すると障害時の調査手段が消える。移行ツールの D 欄解釈の検証点。
場所: RBKEN02C、RAAUP00C。


## J. 潜在バグ系 — 未発火欠陥(6)

本カテゴリは他と軸が異なる: **メインフレーム上でも本物の欠陥**だが、現行データが
引き金条件を満たしたことがなく「正しく動作しているように見える」コード群。
現行データセットは引き金を踏まないことを生成時に保証し(dirty_ledger とは別管理)、
引き金を踏む**挑発データセット(app/data/trigger/)** を別添する。
移行演習では (1) 静的解析で検出できるか、(2) 検出した際に修正するか
バグ互換で凍結するかの判断と根拠、(3) 挑発データ投入時の移行前後の挙動差
(黙った桁落ち vs 例外)を採点する。個別の埋込位置・引き金条件・MF での
発火時挙動は **docs/07_senzai_bug_daicho.md(潜在バグ台帳)** に全件記録する。

**J-01 桁あふれ未防御(ON SIZE ERROR なし)** ★★★
現象: 累計・カウンタへの ADD/COMPUTE に ON SIZE ERROR がなく、理論上あふれ得るが現行データの最大値では届かない。発火時は**高位桁が黙って切り捨てられ**、もっともらしい誤値で処理が続く。
実証済: RUTLDTC の NS 機能(年カウンタ 9(4))。通算日 9999999 で RC=00 のまま「9358年」を返す(tests/TESTJ01)。他: RBRYO01C 大口需要家の年間累計域。

**J-02 桁ずれ・切捨て MOVE** ★★
現象: 長い項目→短い項目への MOVE(X(40)→X(30)、9(7)→9(5))。現行データは常に短く収まるため無害。発火時は末尾/高位が静かに欠落。
場所: RBRYO02C 請求書宛名編集、RBKEN02C。

**J-03 添字範囲外アクセス(SSRANGE なし)** ★★★
現象: OCCURS 12 の月テーブルに、境界条件で 13 になり得る添字でアクセス。現行データでは添字は常に 1〜12。発火時は**隣接する作業域を静かに読み書き**し、アベンドせず別項目を破壊する(SSRANGE/CHECK なしの既定動作)。
移行: 配列境界チェックのある言語では即例外 — 「移行したらアベンドするようになった」の典型例。
場所: RBKEN02C(検針月+1 の先読み)、RBGET01C。

**J-04 突合マッチングの重複キー未考慮** ★★
現象: ソート済み2ファイルの突合(バランスライン)が「キー一意」を暗黙前提とし、重複キー到来時に読み飛ばし/二重処理する。現行データはユニーク保証済みのため未発火。
場所: RBSHU01C(収納突合)、RBTAI01C。

**J-05 ゼロ除算未防御** ★★
現象: 日割り計算の除数(使用日数)がゼロになり得る(同日開始・解約)が、現行データに該当契約が存在しない。発火時は MF でもデータ例外(S0Cx)でアベンド — つまり**両環境で落ちるが、落ち方と復旧手順が異なる**。
場所: RBRYO01C(日割り)、RBAUD01C(平均計算)。

**J-06 境界条件の off-by-one** ★★★
現象: 判定の > と >= の取り違え(デマンド超過率ちょうど 100.00% の扱い)。現行データに境界値ちょうどが存在せず未発火。B-10(浮動小数点閾値)が表現誤差の問題であるのに対し、こちらは**固定小数点でも存在する論理誤り**。
移行: 仕様書化の際に「100%超過で警報」と「100%以上で警報」のどちらを意図と認定するかの判断問題。挑発データにちょうど 100.00% を含める。
場所: RAAUP01C(デマンド判定)、RBRYO01C(段階境界)。


## K. 規格・世代混在系(6)

COBOL-74(OS/VS COBOL)世代の作法と COBOL-85(VS COBOL II 以降)世代の作法が
同一システム・同一プログラム内に地層をなすパターン。RUTLDTC が実例
(74 様式主体+85 規格の参照修飾が同居)。

**K-01 74規格・廃要素の残存** ★
現象: LABEL RECORDS ARE STANDARD / DATA RECORDS 句、SECTION の優先番号(セグメンテーション)、77 レベル多用など、現行コンパイラでは廃要素(obsolete)警告つきで通る 74 作法が残存。
MF: 警告は 30 年間無視されてきた(「警告ゼロ」規約は後年のもの)。
移行: パーサ・変換器が廃要素で停止/警告を仕様と誤読。旧版 RBRYO00C に高濃度で埋込む。
場所: RBRYO00C、各所に薄く。

**K-02 現在日付取得の世代混在** ★★★
現象: ACCEPT ... FROM DATE(**2桁年 YYMMDD**)、ACCEPT ... FROM DATE YYYYMMDD、FUNCTION CURRENT-DATE の3方式が併存。2桁側は E-03 の窓割りに接続。
MF: 各方式とも正しく、混在しても現行は矛盾しない。
移行: 現在日付の一括置換で桁が合わずズレる。テスト用の日付固定(タイムマシン)機構が方式ごとに効いたり効かなかったりし、**現新比較の実行日依存差**を生む。
場所: RASGN00C(FUNCTION)、RBTAI01C(2桁 ACCEPT)、RBGET01C。

**K-03 同一機能の世代別二重作法** ★★
現象: EVALUATE と GO TO DEPENDING、INITIALIZE と MOVE SPACES/ZEROS 連打、インライン PERFORM と外部段落 PERFORM、SEARCH と番兵つき自前ループ — 同じことを別世代の書法で行う箇所が同一プログラム内に併存。
MF: どちらも正しい。改修年代の考古学的地層。
移行: 変換器が二様式へ別々の変換規則を適用し、**同一業務ロジックが異なる実装**になって保守が二重化。リファクタリング統一の判断材料(どちらへ寄せるか)も問う。
場所: RBRYO01C、RAKYK02C。

**K-04 ソースとロードモジュールの不整合疑い** ★★★
現象: 翻訳記録簿(コンパイルリスト日付・ロード日付)とソース最終修正日が矛盾する資産を混在させる — ソースが新しいのにロードが古い(未反映修正)、ロードが新しいのにソースが古い(ソース紛失・別ライブラリ翻訳の疑い)。
MF: 動いているのはロードモジュールであり、日々の運用に支障はない。
移行: **「どのソースが真実か」**という移行の大前提が崩れる。ソース起点の移行は現物挙動と食い違い、現新比較で原因不明の不一致となる。翻訳記録簿を資産として提供し、突合演習とする。
場所: 翻訳記録簿(app/data/compile_ledger.csv)、矛盾側に RBSTM02C・RBRYO00C を指定。

**K-05 ピリオド終端とスコープ終端子の混在** ★★
現象: 同一段落内に END-IF 様式(85)とピリオド終端様式(74)が混在し、さらにインデントが実際の制御範囲と一致しない箇所を含む(実行は正しい)。
MF: コンパイラは正しく解釈する。人間は誤読してきたが、動くので直されなかった。
移行: 人間・AI の目視読解が範囲を誤る視覚的トラップ。C-05(NEXT SENTENCE)の誤飛越と複合すると仕様書化の誤りに直結。
場所: RAKYK02C、RBKEN01C。

**K-06 化石ソース(現行コンパイラで翻訳不能な旧規格メンバ)** ★★
現象: 1998 年以降翻訳されていない OS/VS COBOL 様式のメンバ(NOTE、EXAMINE、TRANSFORM、READY TRACE を含む)が PDS に残存。I-04(未参照)の極端形で、**現行コンパイラでは翻訳が通らない**。
MF: 誰も触らないので問題にならない。資産台帳には載っている。
移行: 一括変換パイプラインがこのメンバで停止する/対象外の判断根拠(翻訳可否・参照有無)を記録できるかを問う。
場所: app/cbl/fossil/RYOKYU68.cbl(翻訳対象外を明示、品質ゲートのコンパイル対象からも除外)。

## 採点の使い方(TSUBAME-BENCH 方式)

1. **検出率**: カタログ非開示で解析し、85パターン中いくつを指摘できたか(★別集計)。
2. **移行正答率**: 各パターンについて「移行後も MF と同一の業務結果」を再現できたか。
   C-05/E-05 のような「正しく動く誤り」は、**凍結(バグ互換)か修正か**の判断と
   その根拠の明示までを正答条件とする。
   J カテゴリ(潜在バグ)は挑発データセット投入時の移行前後挙動差
   (黙った桁落ち/隣接破壊 vs 例外)まで含めて採点する(docs/07 台帳参照)。
3. **性能**: 検針72万件の料金計算(D030)と30分値リプレイ(オーソリ相当)の
   スループットを移行前後で比較(データ規模は 04_data_sekkei.md)。
